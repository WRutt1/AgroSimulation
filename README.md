## 🌾 Симулятор Агрофермы (AgroSimulation)

Текстовая пошаговая стратегия управления сельскохозяйственными угодьями. Вы играете за управляющего фермой, планируете посевы, следите за созреванием культур и собираете урожай, стараясь приумножить свой бюджет.

### 🎮 Как играть

Процесс игры разбит на дни (ходы). Вы не можете выполнять действия мгновенно. Сначала вы планируете задачи на день, а затем переходите к следующему дню, чтобы техника поехала их выполнять.

### 📋 Основное меню

1. **Добавить задачу:** Открывает подменю для планирования работ на поле. Вы можете выбрать:
* **1 (Посев):** Укажите точное название поля (например, **Поле_А**) и точное название культуры с большой буквы (например, **Пшеница**).
* **2 (Сбор):** Укажите точное название поля, на котором созрел урожай.


2. **След. день:** Завершает текущий ход. Выполняются все запланированные задачи (списываются деньги и топливо), а посаженные культуры прибавляют 1 день к своему росту.
3. **История поля:** Введите ID поля, чтобы получить полную сводку всех сельскохозяйственных работ, которые на нем проводились.
4. **Выход:** Завершение симуляции.

### 🚀 Пример быстрого старта

1. Нажмите **1** -> **1**, введите **Поле_А** и культуру **Пшеница**.
2. Нажмите **2** (След. день). Спишутся деньги, и поле поменяет статус на «засеяно».
3. Нажимайте **2**, пока статус поля не изменится на «созрело» (занимает 3 игровых дня).
4. Нажмите **1** -> **2** и введите **Поле_А** для планирования сбора урожая.
5. Нажмите **2** (След. день), чтобы собрать урожай на склад.

### 💡 Важные нюансы

* **Регистр важен:** Вводите названия полей (`Поле_А`, `Поле_Б`) и культур (`Пшеница`, `Кукуруза`, `Ячмень`) в точности так, как они отображаются на экране.
* **Очередь задач:** Вы можете запланировать сразу несколько задач для разных полей в один день. Все они будут выполнены одновременно при переходе к следующему дню.
* **Ограничения:** Если у вас закончится бюджет или топливо в тракторе, запланированная задача не будет выполнена и игра выдаст предупреждение.

### 1. Архитектура

Архитектуру можно разделить на три логических слоя:

* **Слой представления (UI/CLI):** Отвечает за общение с пользователем через консоль. Реализован внутри методов `show_status()`, `add_task()` и основного цикла `run()`.
* **Слой бизнес-логики (Engine):** Ядро симуляции. Находится в классе `AgroSimulation`. Отвечает за течение времени (`next_day()`), проверку условий (хватает ли денег или топлива) и выполнение отложенных задач (`process_tasks()`).
* **Слой данных (Entities & Structures):** Набор классов-сущностей (`Field`, `Machine`, `Crop`, `Warehouse`) и кастомных структур данных (`LinkedList`), которые хранят состояние игры.



### 2. Структура классов и их связи (ООП)

В проекте используются классические паттерны связи объектов: **Композиция** (жесткая связь, часть не существует без целого) и **Агрегация** (мягкая связь, объекты могут существовать отдельно).

| Класс | Роль в системе | Ключевые атрибуты | Отношения с другими классами |
| --- | --- | --- | --- |
| **AgroSimulation** | Главный контроллер | `day`, `budget`, `task_queue` | **Композиция:** жестко владеет складом, полями и техникой. |
| **Warehouse** | Склад фермы | `storage` (Словарь) | Подчиняется главному контроллеру. |
| **Field** | Игровое поле | `area`, `status`, `days_growing` | **Агрегация:** может временно содержать объект `Crop`.<br>

<br>**Композиция:** жестко владеет `LinkedList`. |
| **Crop** | Данные о культуре | `name`, `grow_days`, `yield_rate` | Справочная сущность, передается полю при посеве. |
| **Machine** | Техника (Трактор) | `fuel`, `wear` | Автономная сущность, выполняет работу и тратит свои ресурсы по запросу движка. |
| **LinkedList** | Журнал истории | Узлы `Node` | Жестко привязан к конкретному экземпляру `Field`. |



### 3. Интеграция алгоритмов и структур данных


* **Очередь задач (`collections.deque`):**
* *Где используется:* В главном классе (`self.task_queue`).
* *Зачем:* Реализует паттерн «Отложенное выполнение» (Command queue). Пользователь добавляет задачи, а движок выполняет их по принципу FIFO (первым пришел — первым ушел) только при наступлении следующего дня.


* **Словарь/Хэш-таблица (`dict`):**
* *Где используется:* На складе (`self.storage`).
* *Зачем:* Обеспечивает мгновенный доступ к ресурсам (Сложность O(1)). Нам не нужно перебирать весь массив ресурсов, чтобы найти Пшеницу, мы обращаемся к ней напрямую по ключу.


* **Связный список (`LinkedList`):**
* *Где используется:* В истории поля (`self.history`).
* *Зачем:* Идеально подходит для ведения текстовых логов. Добавление новой записи в конец списка происходит эффективно, а размер истории динамичен и не требует выделения фиксированного массива памяти.


* **Быстрая сортировка (`Quick Sort`):**
* *Где используется:* В функции `quick_sort_fields()`.
* *Зачем:* Отвечает за UX (пользовательский опыт). Применяется перед каждым выводом статуса на экран, чтобы поля всегда выводились отсортированными по убыванию площади.


* **Бинарный поиск (`Binary Search`):**
* *Где используется:* В функции `binary_search_catalog()`.
* *Зачем:* Оптимизирует поиск по справочнику цен магазина. Если бы каталог состоял из десятков тысяч позиций, поиск нужных семян всё равно занял бы миллисекунды (Сложность O(\log n)).


### 4. Жизненный цикл программы (Execution Flow)

1. **Инициализация:** Выделяется бюджет, создаются пустые поля, инициализируется склад и техника.
2. **Ожидание ввода (Idle):** Система выводит интерфейс через `show_status()` и ждет команду пользователя.
3. **Буферизация команд:** Если игрок выбирает действие, создается словарь с описанием задачи и помещается в правую часть очереди. Само состояние игры пока не меняется.
4. **Тик симуляции (Tick):** При команде «След. день»:
* Обновляются счетчики роста у всех засеянных полей.
* Запускается цикл извлечения задач из левой части очереди.
* Проверяются бизнес-правила (хватает ли денег на семена, есть ли топливо в баке).
* Состояние системы изменяется (списывается бюджет, меняется статус поля, пополняется склад).
